Le modèle relationnel & l'ordre logique

Tables, clés, et l'ordre FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY
L'idée du chapitre SQL ne s'exécute pas dans l'ordre où on l'écrit. On écrit SELECT en premier, mais la base le traite presque en dernier. Connaître l'ordre d'exécution logique est la clé n°1 pour prédire un résultat à l'aveugle (et pour comprendre les messages d'erreur).
1.1 — Tables, lignes, colonnes, clés
En clair Une table, c'est un tableau Excel : des colonnes (les champs, ex. name) et des lignes (les enregistrements, ex. « Alice, 22 »). La clé primaire est le numéro unique de chaque ligne (comme un numéro de sécurité sociale). La clé étrangère est un champ qui pointe vers la clé d'une autre table (ex. enrollments.student_id pointe vers students.id) : c'est ce qui relie les tables.
Fiche · Vocabulaire relationnel
  • Table = relation ; ligne = enregistrement ; colonne = attribut.
  • Clé primaire (PK) : identifie chaque ligne de façon unique (jamais NULL, jamais en double).
  • Clé étrangère (FK) : référence la PK d'une autre table. Ex. enrollments.student_id → students.id.
  • Intégrité référentielle : une FK doit pointer vers une ligne existante (ou être NULL).
1.2 — L'ordre d'exécution logique (LA fiche clé)
En clair Imagine une chaîne de montage. D'abord on va chercher la table (FROM), on jette les lignes non voulues (WHERE), on fait des paquets (GROUP BY), on jette des paquets entiers (HAVING), on choisit les colonnes à afficher (SELECT), on trie (ORDER BY), puis on coupe (LIMIT). Tu écris SELECT en haut, mais l'ouvrier le fait presque à la fin !
Fiche clé · Ordre d'exécution
1. FROM / JOIN  →  2. WHERE  →  3. GROUP BY  →  4. HAVING  →  5. SELECT  →  6. ORDER BY  →  7. LIMIT

Conséquences pratiques (sources d'erreurs aux examens) :

  • Un alias défini dans le SELECT n'existe pas encore au moment du WHERE (SELECT vient après) — mais il existe dans ORDER BY.
  • WHERE filtre des lignes brutes ; HAVING filtre des groupes (après agrégation).
  • On ne peut pas mettre une fonction d'agrégat (COUNT, AVG…) dans un WHERE.
-- Squelette complet, dans l'ordre où on l'ÉCRIT
SELECT   colonnes, AGG(...)      -- 5
FROM     table                  -- 1
WHERE    condition_ligne        -- 2
GROUP BY colonnes               -- 3
HAVING   condition_groupe       -- 4
ORDER BY colonnes               -- 6
LIMIT    n;                     -- 7
QCM 1

Quelle clause est exécutée en premier ?

B. FROM (on récupère la table) avant tout. SELECT n'arrive qu'en 5ᵉ position, malgré sa place en haut du code.
QCM 2

Pourquoi SELECT age AS a FROM students WHERE a > 21 échoue ?

C. WHERE (2) s'exécute avant SELECT (5), donc l'alias « a » n'est pas encore défini. Il faut écrire WHERE age > 21.
QCM 3 · prédis le résultat

Combien de lignes renvoie SELECT * FROM students; ?

A. 4 étudiants (Alice, Bob, Carol, David). SELECT * sans filtre = toutes les lignes.
Exercice corrigé — l'ordre logique en action

Lis cette requête et dis ce qu'elle renvoie : SELECT name FROM students WHERE age >= 22 ORDER BY age DESC;

Voir la solution

FROM students → 4 lignes. WHERE age ≥ 22 → Alice (22), Bob (25). Carol (NULL) exclue, David (20) exclu. ORDER BY age DESC → Bob, puis Alice. SELECT name.

Résultat : Bob, Alice (2 lignes, dans cet ordre).

Filtrer & les règles NULL

SELECT, WHERE, opérateurs, et la logique à trois valeurs de NULL
L'idée du chapitre NULL n'est pas zéro ni une chaîne vide : c'est « valeur inconnue ». Toute comparaison avec NULL donne UNKNOWN (ni vrai ni faux), et le WHERE ne garde que ce qui est VRAI. C'est la première source de bugs « invisibles » aux examens.
2.1 — SELECT & WHERE
En clair SELECT = « quelles colonnes je veux voir ». WHERE = « quelles lignes je garde ». Pense au WHERE comme à un videur à l'entrée : il pose une question à chaque ligne et ne laisse entrer que celles qui répondent VRAI. Si la ligne répond « je ne sais pas » (à cause d'un NULL), le videur la refuse.
Fiche · Opérateurs du WHERE
OpérateurSens
=, <>, <, >, <=, >=comparaisons
AND, OR, NOTcombiner des conditions
BETWEEN a AND bintervalle (bornes incluses)
IN (...)appartient à une liste
LIKE 'A%'motif (%=n'importe quoi, _=1 car.)
IS NULL / IS NOT NULLtester la présence de NULL
2.2 — Les règles NULL (le piège n°1)
En clair NULL = « case vide, on ne sait pas ce qu'il y a dedans ». Donc « age = NULL ? » n'a pas de réponse : peut-être oui, peut-être non → UNKNOWN. Même NULL = NULL donne UNKNOWN (deux inconnues ne sont pas « égales »). Pour tester une case vide, il faut un test spécial : IS NULL.
Fiche clé · Logique à 3 valeurs
  • Toute comparaison avec NULL (=, <, >…) donne UNKNOWN, jamais TRUE/FALSE.
  • Le WHERE ne garde que les lignes TRUE (UNKNOWN est rejeté, comme FALSE).
  • Pour tester un NULL : IS NULL / IS NOT NULL (jamais = NULL).
  • NULL en arithmétique « contamine » : NULL + 5 = NULL, NULL || 'x' = NULL.
  • Les agrégats ignorent NULL (sauf COUNT(*)) — voir ch. 3.
Piège : WHERE age <> 22 n'inclut pas Carol (age NULL), car NULL <> 22 = UNKNOWN, pas TRUE. Beaucoup d'étudiants l'oublient.
-- Gérer proprement les NULL
SELECT name, COALESCE(age, 0) AS age_ou_zero
FROM students
WHERE age IS NOT NULL;     -- exclut Carol explicitement
QCM 1 · prédis le résultat

Combien de lignes : SELECT * FROM students WHERE age > 21; ?

C. Alice (22) et Bob (25). David (20) ne passe pas, et Carol (NULL) non plus car NULL > 21 = UNKNOWN. → 2 lignes.
QCM 2 · prédis le résultat

Combien de lignes : SELECT * FROM students WHERE age <> 22; ?

D. Bob (25) et David (20). Alice (22) est exclue (=22), et Carol (NULL) aussi car NULL <> 22 = UNKNOWN. → 2 lignes. Le piège classique !
QCM 3

Pour récupérer Carol (age inconnu), j'écris :

A. = NULL donne toujours UNKNOWN (0 ligne). Seul IS NULL teste l'absence de valeur.
Exercice corrigé — NULL et NOT IN

Que renvoie SELECT name FROM students WHERE age NOT IN (20, 25); ?

Voir la solution

David (20) et Bob (25) exclus. Alice (22) passe. Carol (NULL) : NULL NOT IN (20,25) = UNKNOWN → exclue.

Résultat : Alice seulement (1 ligne). Le NULL disparaît silencieusement, même avec NOT IN.

Agrégation & GROUP BY

COUNT, SUM, AVG, GROUP BY, HAVING — et COUNT(*) vs COUNT(colonne)
L'idée du chapitre Agréger, c'est résumer plusieurs lignes en une seule (un total, une moyenne, un comptage). GROUP BY fait des paquets, et l'agrégat s'applique à chaque paquet. Le piège-roi : COUNT(*) compte les lignes, COUNT(colonne) compte les valeurs non NULL.
3.1 — Fonctions d'agrégation
En clair Une fonction d'agrégation prend une colonne entière et la réduit à un seul chiffre : COUNT compte, SUM additionne, AVG fait la moyenne, MIN/MAX trouvent le plus petit/grand. Détail crucial : elles sautent les cases vides (NULL) — sauf COUNT(*) qui compte les lignes coûte que coûte.
Fiche clé · COUNT(*) vs COUNT(col)
  • COUNT(*) : nombre de lignes (NULL compris).
  • COUNT(colonne) : nombre de valeurs non NULL de cette colonne.
  • COUNT(DISTINCT col) : nombre de valeurs distinctes non NULL.
  • SUM, AVG, MIN, MAX : ignorent les NULL. ⚠️ AVG divise par le nombre de non-NULL, pas par le total.
Sur notre base : COUNT(*) FROM students = 4, mais COUNT(age) = 3 (Carol exclue). Et AVG(age) = (22+25+20)/3 = 22,33 — pas /4.
QCM 1 · prédis le résultat

SELECT COUNT(*), COUNT(age) FROM students; renvoie :

B. COUNT(*) = 4 lignes. COUNT(age) = 3 car l'age de Carol est NULL (non compté).
QCM 2 · prédis le résultat

SELECT AVG(age) FROM students; vaut :

C. AVG ignore le NULL : (22+25+20)/3 = 22,33. Il ne divise PAS par 4. C'est différent de SUM(age)/COUNT(*) !
3.2 — GROUP BY & HAVING
En clair GROUP BY = trier les lignes en paquets qui partagent la même valeur (ex. un paquet par cours). Ensuite l'agrégat compte/additionne dans chaque paquet. HAVING = un deuxième videur, mais qui juge les paquets entiers après comptage (ex. « ne garde que les cours avec ≥ 2 inscrits »), alors que WHERE jugeait les lignes une par une, avant.
Fiche · La règle d'or du GROUP BY

Dans un SELECT avec GROUP BY, chaque colonne du SELECT doit être soit dans le GROUP BY, soit dans un agrégat. Sinon : erreur (ou résultat imprévisible selon le SGBD).

-- Nb d'inscrits par cours, seulement ceux avec >= 2
SELECT course_id, COUNT(*) AS n
FROM enrollments
GROUP BY course_id
HAVING COUNT(*) >= 2;

WHERE filtrerait des inscriptions avant le comptage ; HAVING filtre des cours après comptage.

QCM 3 · prédis le résultat

La requête ci-dessus (GROUP BY course_id HAVING COUNT(*)>=2 sur enrollments) renvoie combien de lignes ?

A. enrollments groupé : course 10 (Math) = 2, course 20 (SQL) = 2. Les deux passent HAVING ≥ 2. History (30) n'apparaît même pas (aucune inscription dans la table). → 2 lignes.
QCM 4

Pour garder seulement les groupes ayant COUNT(*) > 1, on utilise :

D. Un agrégat ne peut pas aller dans WHERE (qui s'exécute avant le regroupement). On filtre les groupes avec HAVING.
Exercice corrigé — moyenne par étudiant

Que renvoie SELECT student_id, AVG(grade) FROM enrollments GROUP BY student_id; ?

Voir la solution

Paquets par student_id : 1 → notes 15,12 → AVG 13,5. 2 → note 8 → AVG 8. 3 → note NULL → AVG sur 0 valeur = NULL.

Résultat : (1 ; 13,5), (2 ; 8), (3 ; NULL). 3 lignes. David (4) n'apparaît pas (aucune ligne dans enrollments).

3.3 — Approfondissements (niveau Master)
Fiche · Au-delà du GROUP BY de base
  • Sous-requêtes : une requête dans une requête. Ex. étudiants au-dessus de la moyenne d'âge : WHERE age > (SELECT AVG(age) FROM students).
  • Fonctions fenêtre (window) : AVG(grade) OVER (PARTITION BY student_id) agrège sans réduire le nombre de lignes (garde le détail + le résumé côte à côte). Essentiel en data science.
  • CTE (WITH ... AS) : nommer une sous-requête pour lisibilité et réutilisation.
  • RANK / ROW_NUMBER / DENSE_RANK : classements (top-N par groupe).
QCM 5

Qu'est-ce qui distingue une fonction fenêtre (OVER) d'un GROUP BY ?

B. Une window function ajoute le résumé en gardant chaque ligne de détail ; GROUP BY, lui, écrase les lignes d'un groupe en une seule.

4

Jointures

INNER, LEFT, RIGHT, FULL — et les pièges des lignes orphelines
L'idée du chapitre Une jointure recolle deux tables via une clé commune. La grande question d'examen : que deviennent les lignes sans correspondance ? INNER les jette ; LEFT garde toutes celles de gauche (en remplissant par NULL). David (sans cours) et History (sans inscrit) sont là pour tester exactement ça.
4.1 — Les types de jointures
En clair Tu as deux listes et tu veux les apparier (les étudiants ↔ leurs inscriptions). INNER JOIN = ne garde que les paires qui matchent des deux côtés (David, sans inscription, disparaît). LEFT JOIN = garde tout le monde à gauche, et met des cases vides (NULL) là où il n'y a pas de partenaire (David reste, avec des NULL). RIGHT = pareil mais à droite. FULL = tout le monde des deux côtés.
Fiche clé · INNER vs LEFT vs RIGHT vs FULL
JointureGarde…Lignes sans match
INNERseulement les correspondancesjetées des deux côtés
LEFTtoutes celles de gauchegauche gardée (droite = NULL)
RIGHTtoutes celles de droitedroite gardée (gauche = NULL)
FULLtoutes des deux côtésles deux gardées (NULL en face)
SELECT s.name, e.course_id
FROM students s
LEFT JOIN enrollments e ON s.id = e.student_id;
QCM 1 · prédis le résultat

SELECT * FROM students s INNER JOIN enrollments e ON s.id=e.student_id; → combien de lignes ?

C. Une ligne par inscription qui matche : (Alice,10),(Alice,20),(Bob,10),(Carol,20) = 4. David (sans inscription) disparaît en INNER JOIN.
QCM 2 · prédis le résultat

Même requête mais en LEFT JOIN → combien de lignes ?

A. Les 4 lignes de l'INNER + David une fois avec course_id = NULL = 5 lignes. LEFT garde tout le monde à gauche.
4.2 — Jointures + agrégation : le combo qui piège
En clair Quand on compte après un LEFT JOIN, il faut choisir quoi compter. Pour David (qui n'a aucun cours mais qu'on garde grâce au LEFT), COUNT(*) dirait « 1 » (il a bien une ligne, avec des NULL), alors qu'on veut « 0 ». La bonne réponse : compter une colonne de la table de droite, COUNT(e.course_id), qui ne compte pas les NULL → David obtient bien 0.
Fiche clé · LEFT JOIN + COUNT 
-- Nb de cours par étudiant, David inclus avec 0
SELECT s.name, COUNT(e.course_id) AS nb_cours
FROM students s
LEFT JOIN enrollments e ON s.id = e.student_id
GROUP BY s.name;

Résultat : Alice 2, Bob 1, Carol 1, David 0. Si on avait écrit COUNT(*), David afficherait 1 (sa ligne LEFT avec des NULL) — l'erreur classique.

QCM 3 · prédis le résultat

Dans la requête ci-dessus, mais avec COUNT(*) au lieu de COUNT(e.course_id), David affiche :

B. COUNT(*) compte la ligne (même pleine de NULL) → David = 1, faux. COUNT(e.course_id) ignore le NULL → David = 0, correct.
QCM 4 · prédis le résultat

Pour lister les cours sans aucun inscrit (donc History), quelle approche ?

C. On part de courses (gauche), LEFT JOIN vers enrollments, et on garde les cours dont le côté droit est resté NULL = aucun inscrit. History ressort. (« anti-join ».)
Exercice corrigé — anti-join

Écris la requête qui liste les cours sans inscrit, et donne le résultat.

Voir la solution 
SELECT c.title
FROM courses c
LEFT JOIN enrollments e ON c.id = e.course_id
WHERE e.course_id IS NULL;

Math et SQL ont des inscriptions (côté droit non NULL) → éliminés. History n'en a pas → côté droit NULL → gardé.

Résultat : History (1 ligne).

4.3 — Approfondissements (niveau Master)
Fiche · Plus loin sur les jointures
  • CROSS JOIN : produit cartésien (toutes les paires). students × courses = 4×3 = 12 lignes. À éviter par accident (oubli du ON !).
  • SELF JOIN : joindre une table à elle-même (ex. hiérarchies employé/manager).
  • Semi-join / anti-join : EXISTS / NOT EXISTS — tester la présence sans dupliquer les lignes.
  • Ordre de jointure & index : les clés étrangères indexées rendent les jointures rapides ; sans index, coût quadratique.
QCM 5 · prédis le résultat

SELECT COUNT(*) FROM students CROSS JOIN courses; vaut :

D. Produit cartésien : 4 étudiants × 3 cours = 12 paires. C'est ce qui arrive aussi si on oublie la condition ON d'une jointure.
  • INNER jette les orphelins ; LEFT garde la gauche (NULL à droite). David teste ça.
  • LEFT JOIN + COUNT(colonne_droite) pour compter 0 correctement (jamais COUNT(*)).
  • Anti-join (LEFT JOIN + WHERE droite IS NULL) pour « ceux sans correspondance » (History).
  • Oublier le ON → produit cartésien (explosion du nombre de lignes).
  • Toujours se demander : où sont les NULL, où sont les lignes orphelines ?

Une window function OVER(PARTITION BY ...) :

Une fonction d'agrégat dans le WHERE :

A. WHERE s'exécute avant l'agrégation, donc COUNT/SUM/AVG y sont impossibles. On filtre les groupes via HAVING.

Pour trouver les étudiants sans aucune inscription (David) :

P12 · Ch.3+4

Titre du cours et sa note moyenne, pour les cours qui ont au moins un inscrit.

FROM courses c
JOIN enrollments e ON c.id = e.course_id

P13 · Ch.2+3

Âge moyen des étudiants (en gérant le NULL) — explique le diviseur.

P14 · Ch.3 (sous-requête)

Étudiants plus âgés que la moyenne.

Solution

WHERE age > (SELECT AVG(age) FROM students);

Moyenne ≈ 22,33. Bob (25) seul dépasse. (Alice 22 < 22,33 ; Carol NULL exclue.)

P15 · Synthèse

Pour chaque étudiant, son nom et son nombre de cours, triés du plus actif au moins actif, David inclus.

Solution
SELECT s.name, COUNT(e.course_id) AS nb
FROM students s
LEFT JOIN enrollments e ON s.id = e.student_id
GROUP BY s.name
ORDER BY nb DESC;

Alice 2, puis Bob 1 et Carol 1, puis David 0. (LEFT JOIN + COUNT(col) + ORDER BY : la requête « complète » type examen.)
